# Порівняння трьох підходів до побудови RAG: Naive, Sentence-Window та Parent-Child Retrieval

Retrieval-Augmented Generation (RAG) — це архітектура, яка поєднує великі мовні моделі (LLM) з зовнішніми джерелами знань для генерації більш точних та обґрунтованих відповідей. Ключовим аспектом RAG є **стратегія ретриву** — як саме ми розбиваємо документи на чанки (chunks), індексуємо їх та відновлюємо контекст для LLM.

Нижче наведено детальне порівняння трьох основних підходів: **Naive RAG**, **Sentence-Window Retrieval** та **Parent-Child Retrieval** (також відомий як Recursive Retriever).

---

## 1. Naive RAG (Наївний підхід)

### Механізм роботи
Це базовий підхід, який використовується в більшості простих RAG-систем.
1.  **Розбиття документів:** Документ розбивається на фіксовані за розміром чанки (наприклад, по 500 або 1000 символів) з певним перекриттям (overlap).
2.  **Індексація:** Кожен чанк індексується окремо. Векторна емпедінг-модель створює векторне представлення для всього чанка.
3.  **Ретрив:** При запиті система шукає чанки з найбільшою семантичною схожістю до запиту.
4.  **Генерація:** Знайдені чанки передаються LLM як контекст для генерації відповіді.

### Переваги
*   **Простота реалізації:** Легко інтегрується в будь-який фреймворк (LangChain, LlamaIndex).
*   **Швидкість:** Мінімум обчислювальних витрат на етапі індексації та ретриву.
*   **Ефективність пам'яті:** Зберігається лише один чанк на одиницю інформації.

### Недоліки
*   **Втрата контексту:** Якщо важлива інформація розділена між двома чанками, жоден з них не містить повної відповіді.
*   **Проблема з займенниками:** Чанк може містити займенник ("він", "воно"), який не має контексту для розуміння, до якого об'єкта він відноситься.
*   **Шум у векторах:** Великі чанки містять багато "шуму" (непов'язаної інформації), що знижує точність семантичного пошуку.
*   **Обмеження контексту LLM:** Якщо потрібно багато чанків, вони можуть не вміститися в контекстне вікно LLM.

---

## 2. Sentence-Window Retrieval (Ретрив вікна речень)

### Механізм роботи
Цей підхід вирішує проблему втрати контексту, використовуючи двоступеневу стратегію:
1.  **Індексація (на етапі завантаження):**
    *   Документ розбивається на найменші можливі одиниці — **речення** (або короткі абзаци).
    *   Для кожного речення створюється векторний емпедінг.
    *   **Ключовий момент:** До метаданих кожного речення додається текст сусідніх речень (вікно). Наприклад, для речення $i$ зберігається текст речень $i-1, i, i+1$ (або більше, залежно від налаштувань `window_size`).
2.  **Ретрив:**
    *   Система шукає найрелевантніші **речення** (маленькі чанки).
    *   Після знаходження релевантних речень, система використовує метадані, щоб відновити повне **вікно контексту** (сусідні речення).
3.  **Генерація:** LLM отримує розширений контекст (вікно), а не лише одне речення.

### Переваги
*   **Висока точність ретриву:** Речення — це семантично завершена одиниця, що дозволяє точно знайти відповідь.
*   **Збереження контексту:** LLM отримує достатньо контексту для розуміння займенників та зв'язків між ідеями.
*   **Ефективність:** Векторна база даних індексує маленькі об'єкти, що прискорює пошук.
*   **Гнучкість:** Можна налаштувати розмір вікна під конкретну задачу.

### Недоліки
*   **Складність реалізації:** Потрібно правильно керувати метаданими та відновленням вікна.
*   **Витрати на зберігання:** Хоча індексується мало, зберігається більше тексту в метаданих.
*   **Ризик втрати структури:** Якщо вікно перетинає межі абзаців або логічних блоків, контекст може бути розірваний.
*   **Обмеження розміру вікна:** Якщо потрібно більше контексту, ніж може вмістити вікно, система не зможе його надати.

## 3. Parent-Child Retrieval (Ретрив "батько-дитина")

### Механізм роботи
Цей підхід (також відомий як **Recursive Retriever** у LlamaIndex) використовує ієрархічну структуру документів:
1.  **Індексація:**
    *   Документ розбивається на **великі чанки (батьківські)** — наприклад, абзаци або секції.
    *   Кожен батьківський чанк розбивається на **маленькі чанки (дитячі)** — наприклад, речення.
    *   Кожен дитячий чанк має посилання на свого батька (через метадані `parent_id`).
    *   **Важливо:** В векторну базу даних індексується **тільки дитячі чанки** (маленькі одиниці). Батьківські чанки зберігаються окремо (або в іншій базі) для подальшого відновлення.
2.  **Ретрив:**
    *   Система шукає релевантні **дитячі чанки**.
    *   Якщо знайдено кілька дитячих чанків з одного батька, система може **об'єднати** їх у батьківський чанк (або повернути батьківський чанк замість дитячих).
    *   Якщо дитячі чанки з різних батьків, система повертає їх окремо.
3.  **Генерація:** LLM отримує або батьківський чанк (якщо дитячих багато), або набір дитячих чанків з їхнім контекстом.

### Переваги
*   **Адаптивність:** Система автоматично вирішує, чи повертати маленькі чанки, чи великий батьківський контекст.
*   **Глибина контексту:** Дозволяє отримати повний контекст великої секції, якщо запит стосується її частини.
*   **Ефективність:** Індексація маленьких чанків забезпечує точність, а батьківські чанки дають широкі знання.
*   **Масштабованість:** Підходить для великих документів з багатою структурою.

### Недоліки
*   **Складність реалізації:** Потрібно керувати двома рівнями ієрархії та логікою об'єднання.
*   **Витрати на зберігання:** Потрібно зберігати як дитячі, так і батьківські чанки.
*   **Складність налаштування:** Потрібно визначити оптимальний розмір батьківських та дитячих чанків.
*   **Ризик дублювання:** Якщо батьківські чанки перекриваються, може виникнути надлишок інформації.

| Характеристика | Naive RAG | Sentence-Window | Parent-Child |
| :--- | :--- | :--- | :--- |
| **Одиниця ретриву** | Фіксований чанк (500-1000 символів) | Речення | Дитячий чанок (речення) |
| **Контекст для LLM** | Тільки знайдений чанк | Вікно сусідніх речень | Батьківський чанок або набір дитячих |
| **Точність ретриву** | Низька/Середня | Висока | Дуже висока |
| **Якість контексту** | Часто неповний | Добрий (локальний) | Відмінний (глобальний + локальний) |
| **Складність реалізації** | Низька | Середня | Висока |
| **Витрати на пам'ять** | Низькі | Середні | Високі |
| **Швидкість ретриву** | Дуже швидка | Швидка | Середня (залежить від об'єднання) |
| **Найкраще для** | Простих запитів, швидких прототипів | Запитів, що потребують локального контексту | Складних документів, багатозадачних запитів |

## Практичні рекомендації

### Коли використовувати Naive RAG?
*   **Прототипування:** Коли потрібно швидко створити MVP.
*   **Прості документи:** Якщо документи короткі та мають чітку структуру.
*   **Обмежені ресурси:** Коли немає можливості використовувати складніші алгоритми.
*   **Приклад:** Пошук конкретних фактів у коротких документах (наприклад, "Яка дата події X?").

### Коли використовувати Sentence-Window?
*   **Запити з займенниками:** Коли потрібно розуміти, до кого відноситься "він", "вона", "воно".
*   **Локальний контекст:** Коли відповідь знаходиться в невеликому сегменті тексту.
*   **Баланс між точністю та швидкістю:** Коли потрібна краща якість, ніж у Naive RAG, але без складності Parent-Child.
*   **Приклад:** Відповіді на питання про конкретні абзаци або речення в наукових статтях.

### Коли використовувати Parent-Child?
*   **Складні документи:** Коли документи мають багато рівнів структури (розділи, підрозділи).
*   **Широкий контекст:** Коли потрібно зрозуміти загальну ідею секції, а не лише окремі факти.
*   **Висока точність:** Коли критично важлива якість відповіді (наприклад, у медичних чи юридичних застосунках).
*   **Приклад:** Аналіз довгих звітів, де потрібно зрозуміти зв'язок між різними частинами документа.

---

## Висновок

Вибір підходу до RAG залежить від конкретних вимог вашого застосунку:
*   **Naive RAG** — це базовий рівень, який підходить для простих задач, але часто недостатній для складних запитів.
*   **Sentence-Window** — це золотий середній шлях, який значно покращує якість контексту без надмірної складності.
*   **Parent-Child** — це найпотужніший підхід для складних документів, але він вимагає більше ресурсів та уваги до реалізації.

Для більшості виробничих застосунків рекомендується почати з **Sentence-Window**, оскільки він забезпечує значне покращення якості відповідей порівняно з Naive RAG, залишаючись відносно простим у реалізації. Якщо ж ви працюєте з дуже складними документами, **Parent-Child** може бути необхідним для досягнення максимальної точності.

# Який підхід найкращий для production?

**Коротка відповідь:** Для більшості production-систем **Sentence-Window Retrieval** є оптимальним вибором, але з обов'язковим додаванням **гібридного пошуку** та **reranking**. Однак, остаточний вибір залежить від конкретних вимог вашого застосунку.

---

## Детальний аналіз для production

### 1. **Sentence-Window Retrieval — "Золота середина" для production**

#### Чому це найкращий вибір для більшості випадків:

**Переваги для production:**
- **Висока точність:** Дослідження показують покращення **Answer Relevance на 22.7%** та **Groundedness на 38.2%** порівняно з Naive RAG
- **Прогнозована затримка:** Час ретриву стабільний, що важливо для SLA
- **Легка масштабованість:** Проста архітектура, легко моніторити та дебажити
- **Низькі витрати:** Менше токенов порівняно з Parent-Child
- **Простота підтримки:** Менше складних метаданих, легше оновлювати

**Вартість:**
- **Індексація:** +15-20% до витрат порівняно з Naive (зберігаються метадані)
- **Ретрив:** +10-15% часу (потрібно відновлювати вікно)
- **Генерація:** -20-30% токенов (точніший ретрив = менше зайвих чанків)

**Коли обирати:**
- Документи середньої складності (10-100 сторінок)
- Запити, що потребують локального контексту
- Бюджет: $0.001-0.005 на запит
- Вимоги до затримки: <2 секунди

---

### 2. **Parent-Child Retrieval — Для складних enterprise-задач**

#### Коли це виправдано в production:

**Переваги:**
- **Максимальна точність:** Найкраща якість для складних документів
- **Адаптивність:** Автоматично обирає між малими та великими чанками
- **Глибина контексту:** Ідеально для юридичних, медичних, фінансових документів

**Недоліки для production:**
- **Висока вартість:** +50-100% витрат на індексацію та зберігання
- **Складність:** Важко відлагоджувати, моніторити, оновлювати
- **Нестабільна затримка:** Залежить від кількості об'єднаних чанків
- **Витрати на пам'ять:** Потрібно зберігати обидва рівні чанків

**Вартість:**
- **Індексація:** +80-120% до витрат
- **Ретрив:** +30-50% часу (логіка об'єднання)
- **Генерація:** -10-20% токенов (кращий контекст)

**Коли обирати:**
- Складні документи з багатою структурою (договори, звіти)
- Високі вимоги до точності (99%+ для compliance)
- Бюджет: $0.005-0.015 на запит
- Команда з досвідом підтримки складних систем

---

### 3. **Naive RAG — Тільки для специфічних випадків**

#### Коли це прийнятно в production:

**Переваги:**
- **Мінімальні витрати:** $0.0005-0.001 на запит
- **Швидкість:** <500мс затримка
- **Простота:** Легко впровадити та підтримувати

**Недоліки:**
- **Низька точність:** 60-70% замість 85-95% у Advanced
- **Високий ризик помилок:** Помилки ретриву = помилкові відповіді
- **Погана масштабованість:** Якість падає зі зростанням бази знань

**Коли обирати:**
- Прості FAQ-системи
- Внутрішні інструменти з низькими вимогами
- MVP або прототипи
- Обмежений бюджет

## Практична рекомендація: **Гібридний підхід з Query Routing**

Для production-систем найкращим рішенням є **комбінований підхід**:

### Архітектура Production RAG:

```
User Query
    ↓
Query Router (класифікатор запиту)
    ↓
┌─────────────────┬─────────────────┬─────────────────┐
│   Simple Query  │ Complex Query   │ Specific Query  │
│   (Naive)       │ (Sentence-Window)│ (Parent-Child)  │
└─────────────────┴─────────────────┴─────────────────┘
    ↓
Hybrid Search (Vector + Keyword)
    ↓
Reranker (BGE/Cohere)
    ↓
Context Enrichment (вікно/батько-дитина)
    ↓
LLM Generation
```

### Переваги гібридного підходу:

1. **Оптимальна вартість:** Використовує дешеві методи для простих запитів
2. **Висока якість:** Застосовує складні методи для складних запитів
3. **Гнучкість:** Легко змінювати правила маршрутизації
4. **Масштабованість:** Можна поступово покращувати систему

---

## Критерії вибору для production

### 1. **Вимоги до точності**
- **<80%:** Naive RAG + Hybrid Search
- **80-90%:** Sentence-Window + Reranker
- **>90%:** Parent-Child + Hybrid Search + Reranker

### 2. **Бюджет на запит**
- **<$0.001:** Naive RAG
- **$0.001-0.005:** Sentence-Window
- **>$0.005:** Parent-Child

### 3. **Вимоги до затримки**
- **<500мс:** Naive RAG
- **500мс-2с:** Sentence-Window
- **2-5с:** Parent-Child

### 4. **Тип документів**
- **Прості (FAQ, статті):** Naive або Sentence-Window
- **Складні (договори, звіти):** Parent-Child
- **Гібридні:** Query Routing

---

## Рекомендації для впровадження

### Етап 1: MVP (1-2 тижні)
- **Підхід:** Naive RAG + Hybrid Search
- **Мета:** Швидкий запуск, перевірка гіпотез
- **Очікувана точність:** 60-70%

### Етап 2: Production Ready (2-4 тижні)
- **Підхід:** Sentence-Window + Reranker
- **Мета:** Висока якість, стабільність
- **Очікувана точність:** 85-90%

### Етап 3: Enterprise Grade (1-2 місяці)
- **Підхід:** Query Routing (Naive + Sentence-Window + Parent-Child)
- **Мета:** Максимальна точність, оптимізація витрат
- **Очікувана точність:** 90-95%

---

## Ключові метрики для моніторингу в production

1. **Context Relevance:** Наскільки релевантний отриманий контекст
2. **Groundedness:** Наскільки відповідь базується на контексті
3. **Answer Relevance:** Наскільки відповідь відповідає запиту
4. **Latency:** Час відповіді (p50, p95, p99)
5. **Cost per Query:** Вартість одного запиту
6. **Hallucination Rate:** Відсоток помилок/галюцинацій

---

## Висновок

**Найкращий вибір для production:**
- **Більшість випадків:** **Sentence-Window Retrieval** + Hybrid Search + Reranker
- **Складні enterprise-задачі:** **Parent-Child** з Query Routing
- **Прості застосунки:** **Naive RAG** з Hybrid Search

**Ключові фактори успіху:**
1. Почніть з простого (Naive), поступово ускладнюйте
2. Обов'язково використовуйте Hybrid Search
3. Додайте Reranker для покращення точності
4. Впровадьте моніторинг та оцінку якості
5. Використовуйте Query Routing для оптимізації витрат

**Рекомендація:** Для більшості production-систем почніть з **Sentence-Window Retrieval** — це забезпечує найкращий баланс між якістю, вартістю та складністю. Якщо з часом виявиться, що цього недостатньо, можна перейти на більш складні підходи.
